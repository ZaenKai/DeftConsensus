# Landing Page Context — Run 2

## Purpose of this revision
Refine landing-page positioning after reviewing project context in `summary.md`.

## Locked Decisions

### 1) Audience positioning
- **Decision**: Position the landing page for business leadership and cross-functional teams across the company, not engineering leadership alone.
- **Why**: The product value proposition is organization-wide voice and alignment in what/how features are built; engineering is the execution path, not the only stakeholder.
- **Alternatives considered**:
  - Engineering-leader-only framing
  - Product-only framing
  - Security/compliance-only framing

### 2) Core pain statement
- **Decision**: Lead with the problem: teams are misaligned on what gets built and why.
- **Why**: This is the highest-level pain that downstream capabilities resolve.
- **Alternatives considered**:
  - Black-box execution concern
  - Delivery speed concern
  - Conflict-resolution concern

### 3) Explainer emphasis
- **Decision**: Prioritize swarm-assisted synthesis/planning from discussions in the explainer modal.
- **Why**: It concretely shows how stakeholder input is transformed into actionable planning artifacts.
- **Alternatives considered**:
  - Deterministic handshake emphasis first
  - One-way mirror/security emphasis first
  - Verification ladder emphasis first

### 4) Sequencing gate
- **Decision**: Define and approve the comprehensive brand guide before finalizing implementation-facing landing-page spec details.
- **Why**: Project rules require brand guide sign-off before frontend feature implementation; doing it now reduces rework and design drift.
- **Alternatives considered**:
  - Finalize page spec first, then brand guide
  - Proceed directly to coding

## Continuity note
Run-1 decisions remain active unless explicitly superseded by this revision.
